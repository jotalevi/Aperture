# Aperture

A fast, local, terminal-native image generation REPL for Apple Silicon. Load your model once, iterate at the prompt, full control over every parameter — with a background job queue and hot reload.

Aperture runs SDXL entirely on your Mac via Metal/MPS — nothing leaves the machine. It keeps your checkpoint resident across the session, so after the initial load you generate at the speed of typing.

## Features

- **Local & private** — runs entirely on your Mac, no cloud
- **Load once, generate many** — model stays in memory across the whole session
- **Background jobs** — queue prompts and keep typing while they render
- **Hot reload** — swap checkpoints without restarting the process
- **Full parameter control** — size, seed, steps, scheduler, guidance, batch count, all inline
- **Photoreal defaults** — low-CFG + DPM++ Karras tuning out of the box
- **Self-bootstrapping** — first run creates the venv and installs dependencies
- **Plain-text config** — token and defaults in a simple local file

## Requirements

- Apple Silicon Mac (M1 or newer)
- Python 3.12 recommended (3.13 has known issues with the ML stack)
- ~10 GB free disk for dependencies + a checkpoint

## Install & run

```bash
git clone <repo-url>
cd aperture
python main.py
```

First run creates `.venv`, installs dependencies, and writes `configs/config.txt`, then drops you into the shell. Subsequent runs launch straight in.

## Setup

Point Aperture at a photoreal checkpoint for best results — base SDXL looks synthetic. Good choices: **RealVisXL**, **Juggernaut XL**, **epiCRealism XL** (download the `.safetensors` from Hugging Face or Civitai).

From inside the shell:

```
>>> --conf.hf_token=hf_xxxxxxxxxxxxxxxx
>>> --conf.checkpoint=RealVisXL_V5.0.safetensors
>>> --conf.outdir=./output
>>> --conf.list
>>> --reload
```

## Usage

Type a prompt and press Enter. Inline params go at the **start** of the line:

```
>>> woman by a cafe window, 35mm, natural skin texture, soft window light
>>> --q=4 --size=1024x1024 candid street photo, overcast light
>>> --sketchmode --q=6 testing compositions quickly
>>> --scheduler=eulera --steps=40 --filename=final close-up portrait, freckles
```

### Generation params

| Param | Default | Description |
|-------|---------|-------------|
| `--size=WxH` | `832x1216` | Output dimensions |
| `--seed=N` | from config | Base seed (increments per image in a batch) |
| `--steps=N` | `30` | Denoise steps |
| `--scheduler=NAME` | `dpmpp2m` | `dpmpp2m`, `dpmppsde`, `euler`, `eulera`, `unipc` |
| `--guidance=F` | `5.0` | CFG scale (lower = more natural) |
| `--q=N` | `1` | How many images to generate |
| `--filename=NAME` | timestamp | Output name (`_1.._N` suffix when `q>1`) |
| `--neg="..."` | built-in | Per-shot negative prompt override |
| `--sketchmode` | off | Fast draft: caps steps@12, cfg@3.0 |

Params persist as sticky defaults — you only specify what changes between shots.

## Background jobs

Activate background mode by typing `--jobs` on its own. While active, prompts you enter are queued and generated by a worker thread so you can keep typing. A job list renders just above the input line:

```
  ── jobs ───────────────────────────────────────────
  [#1   | woman by a cafe win… | 1718…_1234.png        | done]
  [#2   | candid street photo… | final.png             |  47%]
  [#3   | close-up portrait,… | 1718…_99.png          | pending]
  ───────────────────────────────────────────────────
>>>
```

| Command | Description |
|---------|-------------|
| `--jobs` | Toggle background mode on/off |
| `--jobs.clear` | Remove finished/errored jobs from the list |

A single worker processes jobs one at a time (generation is GPU-bound). `--q=N` enqueues N seed-incremented jobs. On exit, Aperture waits for the in-flight job so you never get a half-written PNG.

## Hot reload

`--reload` reloads `configs/config.txt` and rebuilds the pipeline in place, so a changed `--conf.checkpoint` takes effect without restarting:

```
>>> --conf.checkpoint=Juggernaut_XL_v9.safetensors
>>> --reload
  loading Juggernaut_XL_v9.safetensors ...
  ready in 14.3s
  reload complete.
```

Queued jobs drain before a reload runs. On 16 GB machines, swapping into a large checkpoint may briefly hold both models in memory — a process restart is the safe fallback if you hit a memory error.

### Config commands

| Command | Description |
|---------|-------------|
| `--conf.KEY=VALUE` | Set & persist a config value |
| `--conf.list` | Show current config (token masked) |
| `--conf.path` | Print config file location |

Keys: `hf_token`, `checkpoint`, `outdir`, `seed`, `steps`, `scheduler`, `guidance`, `default_size`

### Shell commands

| Command | Description |
|---------|-------------|
| `--help` / `--h` | Show the manual |
| `--reload` | Rebuild pipeline from current config |
| `Ctrl+C` | Exit (waits for the active job) |

## Tips for realism

- **Checkpoint matters most** — a photoreal fine-tune beats any param tuning on base SDXL.
- **Keep CFG low** (4–5.5). High guidance creates the oversaturated "AI sheen."
- **Prompt like a photographer** — describe lens, lighting, and film stock.
- **Generate at native resolution** (832×1216 portrait, 1024×1024 square) and upscale after.
- **Use `--sketchmode`** to find a composition fast, then re-run without it for the final.

## Testing & benchmarking

`tests.py` runs unit tests over the core logic and benchmarks pipeline load + generation, writing a report file.

```bash
python tests.py                 # unit tests + benchmarks (loads the model)
python tests.py --no-bench      # unit tests only, no model load
python tests.py --bench-only    # benchmarks only
python tests.py --runs 5        # number of generation iterations
python tests.py --out myreport.txt
```

The report includes system details (chip, RAM, MPS availability, network latency), unit-test pass/fail, load timing, generation mean/min/max, captured logs and errors, and a non-reversible salted run id.

## Share your benchmark results

Help build a cross-device performance picture by contributing your report. It's a one-file PR — no code changes needed.

1. **Fork** this repo.
2. **Generate a report:**
```bash
   python tests.py
```
   With no `--out`, the report is written to `bench/<run_id>.txt` automatically — the run id is the non-reversible hash, so the filename is already unique and collision-safe. Don't rename it.
   
3. **Commit only that one file** under `bench/`:
```bash
   git add bench/<run_id>.txt
   git commit -m "bench: add results"
   git push
```
   (tip: `git add bench/` picks up the new report without you typing the long hash)
   
4. **Open a pull request.** PRs that *only add* a single file under `bench/` are **merged automatically** by CI — no review wait. Anything touching code, or modifying/deleting existing files, goes to normal review.

### What auto-merges

The auto-merge check passes only when **every** condition holds:
- the PR adds files (status `A`) and makes **no modifications or deletions**;
- **every** changed path is inside `bench/`;
- the file is plain text.

If any check fails the PR stays open for a maintainer. This keeps the fast path safe — a benchmark drop-off can't smuggle in a code change.

**Note:** `configs/config.txt` stores your HF token in plain text. It's gitignored — keep it that way, never commit it.
